Booting Is Deployment: MAAS, FOG, and the Problem That Disappeared

July 2026 · POC writeup · full source in /poc

Somewhere in the POC repo are two complete, working runbooks for deployment systems I don't use. I evaluated Canonical MAAS and FOG Project properly — installed them, imaged real machines, wrote down the gotchas — and then benched both. Not because either lost a bake-off. Because the RAM-boot pipeline deleted the problem they solve. The most satisfying tooling decision I've made all year wasn't picking a tool; it was noticing I could stop needing one.

The problem as I originally framed it

The plan, before RAM-boot was the plan, went: build one golden disk image, clone it onto N mismatched physical boxes, then specialize each clone — hostname, identity, the grub argument that turns on the stateless overlay. Cloning disks and specializing clones is precisely the job MAAS and FOG exist for, so I evaluated both. Both worked. That's worth saying clearly before I explain why neither is running: these are good tools, and the runbooks in 06-deployment/ got real machines from powered-off to domain-joined.

MAAS: the smooth road

MAAS was the pleasant surprise. Three things lined up almost eerily well with the architecture. It knows every machine by MAC and lets you name machines — which maps one-to-one onto the hosts.csv that drives keytab staging. It injects cloud-init user-data natively, so the specialization file in the repo just gets pasted into the deploy form. And it will write a completely foreign disk image verbatim: 

qemu-img convert -O raw golden.qcow2 golden-sssd.img
gzip golden-sssd.img
maas $PROFILE boot-resources create \
  name='custom/suse-sssd-stateless' \
  architecture='amd64/generic' \
  filetype='ddgz' \
  content@=golden-sssd.img.gz

ddgz means "gunzip this and dd it onto the disk" — MAAS (an Ubuntu-flavored tool) neither knows nor cares that there's a SUSE system inside. And the lifecycle story was genuinely elegant: release a machine, redeploy it with a new image version, and the AD computer object and the vaulted keytab are untouched — the node reclaims its identity on first boot with no re-join. My runbook calls that "the core payoff of the architecture," and it's the sentence that turned out to matter most, though not the way I expected. Two gotchas worth preserving: if MAAS runs DNS for the subnet it needs forwarders to the AD domain controllers or SSSD's SRV discovery dies, and the MAC in your CSV had better be the NIC that actually PXE-boots.

FOG: the multicast road

FOG's pitch is different: when you need to image many boxes at once, multicast deployment writes them all simultaneously from one stream. The trade-off is that FOG has no native cloud-init, so specialization means a "Snapin" script that runs at first boot. Except — and this is the moment the ground started shifting — while writing that Snapin I realized the minimum viable FOG flow needs zero snapins. A bit-perfect clone boots, and the vault already handles everything a specialization step would do: the fetch script looks up KEYTAB_MAC_<MAC> when the hostname is generic and sets the hostname from HOSTNAME_MAC_<MAC>. The architecture had quietly absorbed per-node specialization before the deployment tool got a chance to perform it. FOG's own gotchas, for the record: resizable single-disk capture wants a plain ext4 root while SUSE defaults to btrfs; multicast floods your switches unless IGMP snooping is configured; and FOG and MAAS fight over the same DHCP scope options, so it's one deployer per subnet, ever.

The realization

Look at what was happening across both evaluations: the specialization section kept shrinking, because vault-assigned identity kept eating it. The deployers were being reduced to exactly one function — writing a disk. Then the RAM-boot decision removed the disk. A deployment system deploys disks; with no disk, there is nothing to deploy, and the remaining "per-node deployment step" is setting PXE-first in the boot order once, at racking. RAM-boot didn't beat MAAS and FOG on features. It dissolved the category they compete in.

the disk road (MAAS / FOG) capture golden image → register every node in the deployer → write each disk (unicast or multicast) → specialize each clone → reboot into the overlay → steady state update or rollback = release + redeploy · per node, per cycle the RAM road build squashfs from the sealed golden → publish images/v2-<date>/ → ln -sfn v2-<date> images/current update or rollback = that symlink · whole fleet, next reboot

Fleet-wide rollout became ln -sfn. Rollback became the same command pointed backwards. Nothing on the disk road competes with that, because the disk road has to visit every machine and the RAM road doesn't visit anything.

Demoted, not rejected

Here's the honest nuance: both runbooks are still in the repo, still working, and deliberately so. They're labeled the fallback path for hardware that can't RAM-boot — not enough memory for a two-and-a-half-gigabyte image plus a working set, or a NIC whose PXE ROM predates hope. And the fallback is cheap to keep precisely because of a design rule from the very first post: the identity flow — vault, keytab fetch, SSSD — is storage-agnostic. A MAAS-deployed node running systemd.volatile=overlay on a real disk executes exactly the same boot-time flow as a diskless one. The deployers were demoted from "the deployment system" to "how the weird hardware gets by," and they handle that role fine.

Two roads I might still take

The evaluation notes end with two forks I marked for later rather than no. The first is openSUSE MicroOS / SLE Micro, whose transactional read-only btrfs root is a genuinely different statelessness model: instead of total amnesia — my design — it keeps persistent state but makes every change an atomic snapshot you can roll back. That's not a variation on this architecture; it's a competing philosophy, and it deserves its own POC rather than a footnote. The second is KIWI NG, SUSE's declarative image builder, which would replace my golden-VM-plus-seal ritual with a described image built from a spec — same rd.live.* boot semantics via its kiwi-live dracut module, and the entire PXE server side (dnsmasq, iPXE, nginx, the versioned current symlink) stays identical. The pattern holds: RAM-boot deleted a problem, and these two are candidates to delete steps. I'll take a deleted step over a new feature most days.

Both runbooks live in 06-deployment/ — maas-deploy.md and fog-deploy.md, complete with the gotchas — and the MicroOS and KIWI notes are in 05-stateless-boot/suse-stateless-notes.md. If you're running a fleet where the deployer earns its keep in ways I haven't hit, I'd genuinely like to know what I'm missing.

Thanks for reading. This came out of a working proof-of-concept — go break it and tell me what I missed. Source
