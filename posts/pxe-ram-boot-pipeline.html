Streaming an OS into RAM: The PXE Boot Pipeline

July 2026 · POC deep dive · full source in /poc

The first post laid out the architecture and the second opened up the keystore, but both of them waved at the actual boot with one ASCII diagram. This post is that diagram at full resolution: how a powered-off box with no disk ends up running a complete OS out of RAM. The whole pipeline turns out to be a DHCP server that never hands out IPs, a thirty-line boot script shared by every machine I own, and a deployment system that is — genuinely — one symlink.

Why RAM-boot beat disk deploy

I kept a disk-based fallback (write the image to disk, mount an overlay on top), and building both taught me why RAM-boot wins for this fleet. Statelessness stops being a discipline and becomes physics: with nothing written to disk, a power cycle is a factory reset, and there is no overlay-escape hatch to audit because there is nothing under the overlay. Just as important for a pile of mismatched hardware: no disk bootloader means no hybrid BIOS/UEFI GRUB gymnastics, no partition geometry, no fstab UUIDs — the only thing that has to understand the hardware is the kernel and its initramfs. And image rollout collapses into "update one file on the PXE server"; the entire fleet is on the new image at its next reboot.

The honest costs: every boot pulls roughly 1.5–2.5 GB over the network, the practical RAM floor is about 8 GB per node (the image lives in memory twice during the copy, then the working set on top), and the PXE server becomes a hard dependency at boot time. I'll come back to why that last one is much less scary than it sounds.

Stage one: the DHCP server that never hands out an IP

The part of PXE that scares people is DHCP, because it sounds like you need to touch the corporate DHCP server — set option 66/67, negotiate with whoever owns it, and explain yourself when something breaks. You don't. dnsmasq has a proxyDHCP mode where it answers PXE clients alongside the real DHCP server: the corporate server keeps handing out IPs exactly as before, and dnsmasq supplements only the boot options. Nobody's scopes get edited. The trimmed config: 

# proxyDHCP: corporate DHCP keeps the leases; we only add boot options
dhcp-range=10.10.20.0,proxy

# architecture detection (RFC 4578 option 93)
dhcp-match=set:bios,option:client-arch,0
dhcp-match=set:efi64,option:client-arch,7
dhcp-match=set:efi64,option:client-arch,9
# iPXE announces itself via user-class — this breaks the chainload loop:
dhcp-userclass=set:ipxe,iPXE

# stage 1: firmware PXE ROM -> tiny iPXE binary over TFTP
dhcp-boot=tag:bios,tag:!ipxe,undionly.kpxe
dhcp-boot=tag:efi64,tag:!ipxe,ipxe.efi
# stage 2: iPXE -> full boot script over HTTP
dhcp-boot=tag:ipxe,http://deploy.poc.lan/boot.ipxe

enable-tftp
tftp-root=/srv/tftp

Three tricks in sixteen lines. The dhcp-match rules read option 93 to tell a dusty BIOS ROM from a UEFI one and serve each the right flavor of iPXE binary. The dhcp-userclass line is the one everyone forgets: once iPXE is running it does DHCP again, and without that tag it would be handed undionly.kpxe a second time and chainload itself forever — iPXE marks its requests with a user-class precisely so the server can hand it the real boot script instead. And TFTP, the slowest protocol still in production use, carries exactly one thing: an iPXE binary measured in kilobytes. Everything heavy moves over HTTP. (One more footgun the setup script handles: dnsmasq is also a DNS server by default and will fight systemd-resolved for port 53, so it gets port=0 to run as pure DHCP-proxy/TFTP.)

Stage two: one boot script for the entire fleet

Everything after the firmware handoff is driven by a single iPXE script, short enough to quote nearly whole: 

#!ipxe
set base http://deploy.poc.lan/images/current

echo === Stateless SSSD RAM-boot ===
echo MAC ${net0/mac}  |  image base: ${base}

kernel ${base}/vmlinuz \
  root=live:${base}/root.squashfs \
  rd.live.ram=1 \
  rd.live.overlay.overlayfs=1 \
  rd.live.dir=/ rd.live.squashimg=root.squashfs \
  rd.neednet=1 ip=dhcp rd.timeout=120 \
  console=tty0 console=ttyS0,115200n8 \
  systemd.hostname=localhost || goto failed

initrd ${base}/initrd.img || goto failed
boot || goto failed

:failed
echo Boot failed — retrying DHCP + chain in 10s
sleep 10
dhcp
chain http://deploy.poc.lan/boot.ipxe

There is no per-node configuration in this file, and that's not laziness — it's the design. Identity doesn't exist at boot time; it's assigned later, in userspace, when the keytab-fetch service asks the vault "who am I?" keyed by NIC MAC. That's also why systemd.hostname=localhost is set explicitly: it's a placeholder that says nobody at this layer names machines. The kernel command line is where the actual magic hides, flag by flag: root=live:http://…/root.squashfs tells dracut the root filesystem is a squashfs at the end of a URL; rd.live.ram=1 copies it fully into RAM and then releases the network hold, so the running system doesn't depend on the HTTP server staying up; rd.live.overlay.overlayfs=1 mounts a tmpfs OverlayFS on top for writes; rd.neednet=1 ip=dhcp brings up whichever NIC answers inside the initramfs; and the dual console= lines mean the same image shows boot output on the local tty and serial-over-LAN, whatever iDRAC or iLO you're staring at. The :failed loop at the bottom is the entire error-handling story: wait ten seconds, re-DHCP, try again, forever. For a stateless node that's correct — there's no local fallback worth falling back to.

Stage three: dracut livenet does the heavy lifting

I didn't write any of the hard part. dracut's livenet and dmsquash-live modules — sitting in the stock dracut package on SLES and openSUSE, no add-on repos — do the HTTP fetch, the RAM copy, and the overlay assembly. Crucially, livenet takes a bare squashfs; some netboot schemes make you wrap the image in an ISO and loopback-mount it, and skipping that step keeps the build pipeline one command shorter and one format dumber.

The trap is the initramfs, and it's the same hostonly trap from the first post wearing different clothes. SUSE's dracut defaults to hostonly mode: it builds an initramfs containing only the drivers the build machine needs. My golden image is built in a VM, so a default initramfs would know about virtio and precisely nothing else — it would boot beautifully in the build VM and hang at "waiting for network" on every real server in the rack. The golden-image build fixes this with one dracut drop-in: 

# /etc/dracut.conf.d/90-ramboot.conf  (written by RAMBOOT=1 image build)
hostonly="no"
add_dracutmodules+=" livenet dmsquash-live overlayfs network base "
add_drivers+=" overlay squashfs loop "
install_items+=" /usr/bin/curl "
compress="zstd"

hostonly="no" pulls in the full generic driver set, and the build then checks its own work — it greps the finished initramfs for the RAID and NIC drivers the fleet actually contains (megaraid_sas, mpt3sas, ixgbe, i40e, mlx5_core…) and warns on anything missing. A driver you forgot is a server that hangs in early boot with no OS to debug from; a warning at build time costs nothing.

Building the artifacts without booting the image

The deploy server turns a sealed golden qcow2 into the three files iPXE references — vmlinuz, initrd.img, root.squashfs — without ever booting the image. build-squashfs.sh uses libguestfs to mount the qcow2 read-only, copies the newest kernel and initramfs out of /boot, and squashes the rest: 

mksquashfs "$MNT" "$OUT/root.squashfs" \
  -comp zstd -Xcompression-level 15 -noappend -no-xattrs \
  -e boot/vmlinuz* -e boot/initrd* -e boot/initramfs* \
  -e proc -e sys -e dev -e run -e tmp

Before it squashes anything, though, it verifies the initramfs it just extracted actually contains the livenet modules: 

for want in livenet dmsquash-live overlayfs; do
  lsinitrd "$OUT/initrd.img" | grep -qi "$want" || MISSING="$MISSING $want"
done
[ -z "$MISSING" ] || { echo "FATAL: initramfs lacks:$MISSING"; exit 1; }

This is my favorite kind of check: it catches the "built the golden image but forgot RAMBOOT=1" mistake as a failed build on the deploy server instead of a rack of servers hanging in early boot. Fail the build, not the fleet. The script finishes by writing SHA256SUMS and printing the RAM-floor arithmetic — squashfs size × 2 plus headroom — so you know immediately whether the smallest boxes can still swallow the new image.

The symlink is the deployment system

Artifacts land in a versioned directory, and one symlink decides what the fleet runs: 

/srv/pxe/images/
├── v1-2026-07-06/   vmlinuz  initrd.img  root.squashfs  SHA256SUMS
├── v2-2026-07-20/   ...
└── current -> v2-2026-07-20

boot.ipxe only ever references images/current/…. Promoting a new image is ln -sfn; rolling back is the same command pointed at the old directory. Every node that boots gets exactly one image, whole — a box either booted v1 or v2, and "partially upgraded" isn't a state that can exist, by construction rather than by promise. The build script stages new versions without promoting by default; you point one node at the staged directory from the iPXE shell, verify it, and then move the symlink for everyone. That two-command deploy-and-rollback story is, honestly, the operational feature that sold me on the whole architecture.

Serving a boot storm

The heavy transfer is plain nginx doing what nginx does at breakfast: 

root /srv/pxe;

sendfile on;
tcp_nopush on;
output_buffers 2 1m;

location / {
    add_header Accept-Ranges bytes;   # iPXE + dracut can resume
}

Big sequential reads of a static file, with range support so an interrupted fetch resumes instead of restarting a two-gigabyte download. Note what's absent: multicast. That's a disk-imaging trick from the FOG/Ghost era; livenet is unicast HTTP, and a 10 GbE NIC on the deploy server handles dozens of simultaneous boots before it breaks a sweat. If the fleet outgrows that, the fixes are config-only — stagger power-on through IPMI, or drop a tiny nginx proxy_cache in each rack and point that rack's boot base at it.

And the dependency question from the top of the post: the PXE server only matters at boot. Once rd.live.ram=1 has done its copy, a running node has no relationship with the deploy server at all. If nginx catches fire, every running box keeps running; the blast radius is "nodes that reboot during the outage boot slowly or wait" — the same graceful failure shape as the keytab vault, and that's not a coincidence, it's the design rule.

Where the pipeline hands off

Seen from the server room floor, the whole conversation looks like this:

node deploy server ─ power on, firmware PXE ROM ─ DHCPDISCOVER (broadcast) → corporate DHCP → "your IP is 10.10.20.41" → dnsmasq (proxy) → "boot file: undionly.kpxe" (option 93: BIOS → undionly.kpxe, UEFI → ipxe.efi) TFTP GET undionly.kpxe → dnsmasq TFTP (tiny — the only TFTP transfer) ─ now running iPXE ─ DHCP again, user-class iPXE → dnsmasq → "chain http://deploy/boot.ipxe" HTTP GET /boot.ipxe → nginx HTTP GET /images/current/vmlinuz + initrd.img → nginx ─ kernel boots, dracut initramfs brings up DHCP ─ HTTP GET /images/current/root.squashfs → nginx (copied fully to RAM) ─ tmpfs OverlayFS over squashfs root → systemd ─ keytab-fetch.service → Infisical vault → hostname + krb5.keytab → SSSD

That last line is where this post ends and the other two begin: systemd starts, the box still thinks it's localhost, and the keytab-fetch service asks the vault who it is by MAC address, installs the keytab into tmpfs, and lets SSSD bring Active Directory logins up. Reboot, and every layer of this stack rebuilds itself from scratch, identically.

Sharp edges

Four things bit me or nearly did. ProxyDHCP is a broadcast trick, so dnsmasq has to sit in the same L2 domain as the nodes (or you get into dhcp-relay); the symptom is a PXE ROM that gets an IP but no boot file, and tcpdump port 4011 on the deploy server settles the question fast. A hang at "waiting for network" inside the initramfs is almost always a missing NIC driver or firmware blob — lsinitrd | grep before blaming the switch. Small-RAM nodes can fill the tmpfs overlay under write-heavy load; journald in the image is already volatile and capped at 64 MB, which turned out to be load-bearing rather than tidy. And Secure Boot bounces UEFI boxes straight back to the PXE menu, because stock iPXE binaries aren't signed — I turned it off for the POC and wrote down the grown-up path (a signed shim + GRUB chain) for whenever this graduates.

Everything quoted here is a real file in /poc/stateless-sssd-poc — start with 08-pxe-ramboot/, whose README has the full comparison table and failure-mode notes, and pxe-server-setup.sh will stand up the serving stack on a spare box in one run. If your fleet PXE-boots a different way — kexec tricks, signed shims, per-rack caches at real scale — I want to hear about it.
