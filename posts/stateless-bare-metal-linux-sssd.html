The Keytab Vault: Diskless Bare-Metal Linux with AD Auth via SSSD

July 2026 · POC writeup · full source in /poc

I wanted physical servers that boot a pristine OS every single time — zero local disk state — and still let AD users log in through SSSD, without a long-lived secret baked into the image. This is the architecture I landed on, why each piece is there, and the sharp edges that only show up once you're dealing with a fleet of mismatched hardware instead of one friendly lab box.

The problem

Stateless infrastructure is easy to reason about when it's a VM or a container: throw it away, boot another one, let the orchestrator hand out identity. Bare metal is the awkward one. There's no hypervisor to give you a fresh disk, the hardware underneath is a grab-bag of NICs, storage controllers, and firmware, and the one piece of state that traditionally can't be stateless — the AD computer account and its Kerberos keytab — is exactly what SSSD needs before a single user can log in.

The obvious move — realm join on every boot — is a trap. It mutates the AD computer object's password each time, litters the domain with machine-account churn, and means the golden image has to carry join credentials. None of that is stateless — it just shoves the state problem into Active Directory and calls it a day.

The approach: join once, fetch forever

So I split identity into two lifecycles that never touch each other:

  • The AD computer object gets created exactly once, out-of-band, by a staging script — never by the host itself.
  • The host fetches a copy of that keytab from a secrets manager on every boot, holds it in RAM only, and throws it away at power-off.

Infisical (self-hosted, runs in Docker) sits in the middle. A staging host runs adcli join once per physical server to create its AD computer object and spit out a keytab, base64-encodes it, and uploads it to Infisical under a key derived from the hostname. I also write a second alias keyed by every NIC MAC address on the box, because a multi-homed server might come up on any cabled port before it has a hostname assigned. The golden image never sees join credentials — it only carries a read-only, project-scoped machine identity that can fetch secrets, not create them.

One line in sssd.conf holds the entire design together: 

ad_maximum_machine_account_password_age = 0

This is the line that will bite you. It disables SSSD's machine-password rotation — miss it and everything works beautifully for 30 days, then SSSD rotates the password out from under the vault, the fetched keytab and the AD object fall out of sync (KVNO mismatch), and every boot after that dies in kinit with Preauthentication failed.

Boot model: streamed into RAM, not written to disk

The boot path I actually care about skips local disks entirely. A PXE ROM hits a dnsmasq proxyDHCP instance (it coexists with the corporate DHCP server, so nobody has to touch the existing scopes), loads a tiny iPXE binary over TFTP, and chains to an HTTP boot script that streams a squashed root filesystem straight into RAM via dracut's livenet module.

Power on → PXE ROM → dnsmasq proxyDHCP (arch detect) → iPXE (TFTP, tiny) → iPXE chains http://deploy/boot.ipxe → kernel+initrd over HTTP → dracut livenet: HTTP GET root.squashfs → copied fully into RAM → tmpfs OverlayFS mounted over the squashfs root → systemd → network up (DHCP) → chrony syncs to DC → keytab-fetch.service: 1. Reads hostname; falls back to boot-NIC MAC 2. POST /api/v1/auth/universal-auth/login → short-lived access token 3. GET /api/v3/secrets/raw/KEYTAB_<HOST> (or KEYTAB_MAC_<MAC>) 4. base64 -d → /etc/krb5.keytab (root:root 0600, tmpfs only) 5. Sanity check: kinit -kt /etc/krb5.keytab '<HOST>$@REALM' → sssd.service starts → AD users can authenticate Reboot → ENTIRE OS discarded from RAM → keytab gone → re-streams pristine image.

Every boot is the same boot. There's no per-node deployment step — booting is deployment, and rollback is me flipping a current symlink on the image server back to the previous SquashFS build. I kept a disk-deploy fallback (OverlayFS over a written-to-disk image) for hardware that can't RAM-boot — not enough RAM, no PXE-capable NIC — but it's exactly that: the fallback, not the design.

Design rules for a fleet, not a lab

One golden image across mismatched hardware only works if the image stops assuming anything about the box it lands on. These are the rules I ended up enforcing:

  • Generic initramfs. Distro dracut defaults to hostonly mode, which strips every driver the build machine doesn't personally need — exactly wrong for a fleet. My build forces full driver inclusion and refuses to finish if common RAID/NIC modules are missing.
  • UUID/LABEL mounts only. The build script refuses to seal an image with /dev/sdX-style fstab entries.
  • No NIC-specific network config. Per-interface network scripts are disabled in favor of NetworkManager's match-all DHCP, and hostname assignment belongs to the vault lookup, not the network stack.
  • Multi-NIC identity resolution. The fetch script tries every physical NIC MAC against the vault, so whichever port someone happened to cable still resolves the right host identity.
  • Dual console output. Both the local tty and serial-over-LAN are wired up, so iDRAC/iLO consoles show boot output on every model.
  • Per-model smoke test before fleet rollout. Gaps get fixed by adding a driver to the single image — never by forking a per-model image. That way lies madness.

What "verified" means here

My pass criterion isn't "it booted once." The verification script checks that the overlay is active, time is synced, the keytab landed with the right principal, kinit -k succeeds, SSSD is healthy, and an AD user resolves via id — and then I reboot the box and run the whole thing again. The failure mode I actually care about is the second boot: state quietly leaking onto disk, a keytab that didn't re-fetch cleanly, or — worst case — a brand-new AD computer object showing up because something ran realm join behind my back. A passing run means the AD object count for a host sits at exactly one across repeated reboots.

The corners I cut (on purpose)

The big one: the machine identity's client secret for read-only vault access is baked into the image. I'm fine with that in a lab; I would not be fine with it in production. The hardening path is written down — conditional access / IP allowlisting on the Infisical side, per-host machine identities instead of one shared fleet identity, TLS with a real certificate instead of a lab endpoint, and rotating the staging credentials once provisioning is done. None of it changes the shape of the architecture; it just tightens the one shortcut I knowingly took.
