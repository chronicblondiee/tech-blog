The Keytab Vault: Diskless Bare-Metal Linux with AD Auth via SSSD

July 2026 · POC writeup · full source in /poc

Physical servers that boot a pristine OS every single time, with zero local disk state, that still authenticate interactive AD users through SSSD — without ever leaving a long-lived secret sitting in the image. Here's the architecture, why each piece exists, and the sharp edges that show up once you have a fleet of mismatched hardware instead of one lab box.

The problem

Stateless infrastructure is easy to reason about when it's a VM or a container: throw it away, boot another one, identity comes from an orchestrator. Bare metal is the awkward case. There's no hypervisor to hand you a fresh disk, the hardware underneath is heterogeneous (different NICs, different storage controllers, different firmware), and the one piece of state that traditionally can't be stateless — the AD computer account and its Kerberos keytab — is exactly what SSSD needs before a single user can log in.

The naive approach is realm join on every boot. That mutates the AD computer object's password each time, litters the domain with machine account churn, and means the golden image itself has to carry join credentials. None of that is stateless — it just relocates the state problem into Active Directory.

The approach: join once, fetch forever

The design splits identity into two lifecycles that never touch each other:

  • The AD computer object is created exactly once, out-of-band, by a staging script — not by the host itself, ever.
  • The host fetches a copy of that keytab from a secrets manager on every boot, holds it in RAM only, and throws it away at power-off.

Concretely: Infisical (self-hosted, Docker) sits between the two. A staging host runs adcli join once per physical server to create its AD computer object and produce a keytab, base64-encodes it, and uploads it to Infisical under a key derived from the hostname — plus a second alias keyed by every NIC MAC address on that box, since a multi-homed server might come up on any cabled port before it has a hostname assigned. The golden image never contains join credentials, only a read-only, project-scoped machine identity that can fetch — not create — secrets.

One line in sssd.conf makes the whole thing hold together: 

ad_maximum_machine_account_password_age = 0

This disables SSSD's default machine-password rotation. Miss it, and the vaulted keytab silently goes stale after 30 days when SSSD rotates the password out from under it — the fetched keytab and the AD object's actual password fall out of sync (KVNO mismatch), and every subsequent boot fails kinit with Preauthentication failed.

Boot model: streamed into RAM, not written to disk

The primary boot path skips local disks entirely. A PXE ROM hits a dnsmasq proxyDHCP instance (it coexists with the corporate DHCP server — no scope changes needed), loads a small iPXE binary over TFTP, and chains to an HTTP boot script that streams a squashed root filesystem straight into RAM via dracut's livenet module.

Power on → PXE ROM → dnsmasq proxyDHCP (arch detect) → iPXE (TFTP, tiny) → iPXE chains http://deploy/boot.ipxe → kernel+initrd over HTTP → dracut livenet: HTTP GET root.squashfs → copied fully into RAM → tmpfs OverlayFS mounted over the squashfs root → systemd → network up (DHCP) → chrony syncs to DC → keytab-fetch.service: 1. Reads hostname; falls back to boot-NIC MAC 2. POST /api/v1/auth/universal-auth/login → short-lived access token 3. GET /api/v3/secrets/raw/KEYTAB_<HOST> (or KEYTAB_MAC_<MAC>) 4. base64 -d → /etc/krb5.keytab (root:root 0600, tmpfs only) 5. Sanity check: kinit -kt /etc/krb5.keytab '<HOST>$@REALM' → sssd.service starts → AD users can authenticate Reboot → ENTIRE OS discarded from RAM → keytab gone → re-streams pristine image.

Every boot is the same boot. There's no per-node deployment step — booting is deployment, and rollback is flipping a current symlink on the image server back to the previous SquashFS build. A disk-deploy fallback (OverlayFS over a written-to-disk image) exists for hardware that can't RAM-boot — insufficient RAM, no PXE-capable NIC — but it's explicitly the fallback, not the design center.

Design rules for a fleet, not a lab

One golden image across mismatched physical hardware only works if it stops assuming anything about the box underneath it:

  • Generic initramfs. Distro dracut defaults to hostonly mode, which strips drivers the build machine doesn't need — exactly wrong for a fleet. The build forces full driver inclusion and fails the build if common RAID/NIC modules are missing.
  • UUID/LABEL mounts only. The build script refuses to seal an image with /dev/sdX-style fstab entries.
  • No NIC-specific network config. Per-interface network scripts are disabled in favor of NetworkManager's match-all DHCP, with hostname assignment left to the vault lookup rather than the network stack.
  • Multi-NIC identity resolution. The fetch script tries every physical NIC MAC against the vault, so whichever port happens to be cabled still resolves the right host identity.
  • Dual console output. Both the local tty and serial-over-LAN are wired up, so iDRAC/iLO consoles show boot output on every model.
  • Per-model smoke test before fleet rollout. Gaps get fixed by adding a driver to the single image — never by forking a per-model image.

What "verified" means here

The POC's actual pass criterion isn't "it booted once." A verification script checks the overlay is active, time is synced, the keytab landed with the right principal, kinit -k succeeds, SSSD is healthy, and an AD user resolves via id — then the box gets rebooted and the whole check runs again. The interesting failure mode isn't first boot, it's second boot: state leaking onto disk, a keytab that didn't get re-fetched cleanly, or — worst case — a new AD computer object appearing because something ran realm join instead of relying on the staged keytab. A passing run confirms the AD object count for a host stays at exactly one across repeated reboots.

Where this is intentionally a POC, not production

The machine identity's client secret for read-only vault access is baked into the image in this build — acceptable in a lab, not in production. The documented hardening path is conditional access / IP allowlisting on the Infisical side, per-host machine identities instead of one shared fleet identity, TLS with a real certificate instead of a lab endpoint, and rotating the staging credentials once initial provisioning is done. None of that changes the core shape of the architecture — it tightens the one deliberate shortcut the POC takes.

Full implementation — golden image build script, Infisical setup, staging script, PXE server config, and the verification script — is in /poc/stateless-sssd-poc in this repo, starting with its own IMPLEMENTATION_GUIDE.md.
