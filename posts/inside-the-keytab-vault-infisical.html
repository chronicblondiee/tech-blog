Inside the Keytab Vault: Infisical as a Boot-Time Keystore

July 2026 · POC deep dive · full source in /poc

In the last post I described diskless bare-metal hosts that fetch their Kerberos keytab from a secrets vault on every boot — and then gave the vault itself, the thing the whole design leans on, about two paragraphs. This is the missing half: what that keystore actually is, how a keytab gets in exactly once, how anonymous hardware gets it back out at every boot, and the parts of the security model I would tighten before letting any of this near production.

Why Infisical and not Vault

The word "vault" primes everyone for HashiCorp Vault, so let me get this out of the way: the keystore is Infisical — an open-source secrets manager that self-hosts as a small Docker Compose stack. No HCL policies, no Shamir key shares, no unseal ceremony. It's a Node app in front of Postgres with a web UI and an HTTP API, which sounds unglamorous until you remember what this component actually has to do: hand one base64 blob to a shell script at boot. I didn't need dynamic database credentials or a PKI engine. I needed a key-value store with real access control and an auth method a curl one-liner could drive.

The honest trade-off: Vault and OpenBao have a much richer identity-attestation story — cert auth, response wrapping, TPM plugins. Infisical's Universal Auth is a client-ID/client-secret pair, morally equivalent to Vault's AppRole and nothing more. That constraint shapes the entire threat model further down, so hold that thought.

The stack: four containers, one stateful volume

Everything runs from one compose file (02-infisical/docker-compose.yml in the POC). Trimmed to what matters: 

services:
  db:
    image: postgres:16-alpine
    volumes:
      - pg_data:/var/lib/postgresql/data

  redis:
    image: redis:7-alpine
    command: redis-server --save "" --appendonly no

  db-migration:
    image: infisical/infisical:latest-postgres
    command: npm run migration:latest

  backend:
    image: infisical/infisical:latest-postgres
    depends_on:
      db-migration:
        condition: service_completed_successfully
    ports:
      - "8080:8080"

volumes:
  pg_data:
  • Postgres is the only thing that holds state. Secrets sit encrypted in the pg_data volume, keyed by an ENCRYPTION_KEY the backend reads from .env at startup. Back up that volume and you've backed up the vault; lose the key and the backup is confetti.
  • Redis is deliberately amnesiac. --save "" --appendonly no — it's cache and queue only, so there is no second place secret material could quietly persist.
  • The migration container is a one-shot gate. The backend refuses to start until migrations finish, which keeps upgrades boring: docker compose pull, up -d, done.
  • It's plain HTTP on :8080. The compose file says so out loud — "Put TLS termination (nginx/caddy) in front for prod." Remember this one for the threat model.

The data model: one flat folder, aliased by everything

Inside Infisical there's a single project (stateless-keytabs), the default prod environment, and one folder: /keytabs. A keytab is a binary file and Infisical stores strings, so every value is base64 -w0 output — nothing clever. What is a little clever is the key naming. Each physical server contributes a small constellation of flat key-value pairs:

stage-keytabs.sh (admin host · once per server) adcli join → web01.keytab → base64 → POST /api/v3/secrets/raw/… auth: keytab-stager · Developer (rw on /keytabs) · revoked after staging ↓ Infisical · project stateless-keytabs · env prod · path /keytabs KEYTAB_WEB01 = <base64 keytab> KEYTAB_MAC_A4BF015E22C1 = same keytab (one alias per cabled NIC) HOSTNAME_MAC_A4BF015E22C1 = web01 state: Postgres (pg_data) only · Redis is deliberately amnesiac ↓ fetch-keytab.sh (every diskless node · every boot) auth: baremetal-boot · Viewer (ro on /keytabs) · baked into the image try KEYTAB_<HOSTNAME> → else KEYTAB_MAC_<MAC> for every physical NIC → /etc/krb5.keytab (root:root 0600, tmpfs) → kinit check → sssd starts

The MAC aliases are the part I'm fondest of. A multi-homed server fresh off the pallet doesn't know its own hostname yet, and whichever NIC someone happened to cable is the one that comes up. So every NIC's MAC becomes a key, all pointing at the same keytab, plus a HOSTNAME_MAC_* entry so the vault can tell the box what it's called. Identity flows from the vault to the host, never the other way: hostname assignment is a keystore lookup, and DHCP option 12 is explicitly not trusted (NetworkManager runs with hostname-mode=none so the network can't rename the box out from under its own keytab).

Two identities, and only one goes in the image

Infisical machine identities are the moral equivalent of service accounts, and the design uses exactly two:

  • keytab-stager — Developer role, read/write on /keytabs. Lives only on the admin host that runs the staging script. The runbook's instruction is blunt: revoke or rotate its client secret the moment staging is done.
  • baremetal-boot — Viewer role, read-only on /keytabs. This is the identity baked into the golden image as /etc/keytab-fetch.env (root-owned, mode 0600): a client ID and client secret, plus the vault URL and project ID. Its access tokens are capped at a 300-second TTL, and the client secret can optionally be pinned to a trusted IP range — the bare-metal subnet — which is the cheapest hardening win in the whole design.

The split means the golden image never carries anything that can create AD objects or write vault secrets. The worst the baked-in identity can do is read keytabs — which is still plenty, and we'll get to exactly how much in a minute.

Keytabs in: join once, upload, shred

The producer side is one script, stage-keytabs.sh, run from an admin box with line of sight to both AD and Infisical. It walks a CSV of hostnames and MACs, and for each host the core move is: 

printf '%s' "$AD_JOIN_PASS" | adcli join "$AD_DOMAIN" \
      --login-user="$AD_JOIN_USER" \
      --stdin-password \
      --computer-name="$SHORT" \
      --host-fqdn="${SHORT}.${AD_DOMAIN}" \
      --domain-ou="$AD_OU" \
      --host-keytab="$KT" \
      --show-details

B64=$(base64 -w0 "$KT")
upsert_secret "KEYTAB_$(echo "$SHORT" | tr 'a-z-' 'A-Z_')" "$B64"
  • adcli join does both halves in one step — it creates the AD computer object in the right OU and emits the keytab. No kinit/ktutil juggling.
  • The staging box never joins anything. --computer-name targets the remote host and --host-keytab points at a per-host file, so the admin host's own identity is untouched. It's a keytab factory, not a domain member doing favors.
  • Uploads are upserts. The script tries POST and falls back to PATCH if the key exists — so re-running it for a host resets the AD machine password, bumps the key version, and overwrites the vault copy in one motion. That's not just idempotency; that's the recovery procedure, and it matters later.
  • Local copies die at the end. The script's parting words are shred -u ./keytabs/*.keytab. The vault copy is the only durable one, on purpose.

Keytabs out: the boot-time fetch

The consumer side is a systemd oneshot wired in front of SSSD. The unit is short enough to quote almost whole: 

[Unit]
Wants=network-online.target chrony.service chronyd.service
After=network-online.target time-sync.target
Before=sssd.service

[Service]
Type=oneshot
ExecStart=/usr/local/sbin/fetch-keytab.sh
RemainAfterExit=yes
TimeoutStartSec=180

[Install]
WantedBy=multi-user.target
RequiredBy=sssd.service
  • The ordering is the whole point. After the network and time sync (Kerberos only tolerates about five minutes of clock skew), strictly before SSSD, and RequiredBy=sssd.service so SSSD can't wander off and start without the fetch at least being attempted.
  • Type=oneshot means systemd won't retry it, so the retry is hand-rolled inside the script: a 30×3-second reachability loop — about ninety seconds of patience for the vault to answer — before anything else happens.
  • Total failure doesn't brick the box. If the vault never answers, the unit fails but boot continues. AD logins are simply unavailable until a reboot fetches successfully; local and console accounts keep working so you can actually debug the thing.

When the fetch does succeed, the installation is deliberately paranoid and deliberately forgiving, in that order: 

umask 077
printf '%s' "$B64" | base64 -d > "${KEYTAB}.tmp"
mv "${KEYTAB}.tmp" "$KEYTAB"
chown root:root "$KEYTAB"; chmod 600 "$KEYTAB"

PRINC="$(echo "$HOSTNAME_SHORT" | tr 'a-z' 'A-Z')\$@${AD_REALM}"
KRB5CCNAME=MEMORY: kinit -kt "$KEYTAB" "$PRINC"

On a RAM-booted node /etc lives in tmpfs, so the keytab never touches a disk — power off and it's gone. The final kinit is advisory by design: a missing vault secret is fatal, but a failed kinit only logs a warning and lets SSSD start anyway. A flaky KDC or a DNS hiccup at second 40 of boot shouldn't take AD auth down for the whole uptime; a genuinely bad keytab gets caught loudly by the verification pass instead.

The honest threat model

Here's the part where I stop being proud of it.

  • One shared read-only credential for the whole fleet, baked into the image. A stateless host has nowhere private to keep a per-host secret — that's practically the definition — so every node authenticates to the vault as the same baremetal-boot identity. Anyone who obtains the golden image has the client secret.
  • Any node can read any node's keytab. The Viewer role covers the entire flat /keytabs folder, and the key names are guessable by construction — they're just hostnames and MACs. Compromise one booted box and you can enumerate the fleet's machine identities.
  • It all rides plain HTTP in the lab. The client secret and the base64 keytab both cross the wire unencrypted on port 8080, at every single boot.

The hardening path is written down in the POC's implementation guide, and none of it changes the shape of the design — which is exactly why I let myself cut these corners. TLS is a one-liner caddy front. The trusted-IP range on the client secret confines credential theft to machines already on the bare-metal subnet. Per-host machine identities with per-host path scoping close the "read your neighbor's keytab" hole, at the cost of the provisioning workflow having to place a distinct secret on each stateless box — which circles back to TPM or hardware-serial attestation as the only really satisfying answer for hardware that owns nothing at power-on. The 300-second token TTL helps less than it looks: captured tokens age out fast, but the client secret is the long-lived thing, and it lives in a squashfs anyone can download from the image server.

The KVNO trap, seen from the vault's side

The first post covered ad_maximum_machine_account_password_age = 0 as the SSSD line that will bite you. From the keystore's side, here's why it's load-bearing: the vault holds a snapshot. A keytab is frozen at a specific key version number (KVNO), and Active Directory expects machine passwords to rotate every 30 days. If SSSD honors that default, the AD object's KVNO advances, the vault's copy silently goes stale, and every subsequent boot fetches a keytab that fails preauthentication. The config template in the golden image doesn't mince words about it: 

# CRITICAL
# Stateless hosts must NEVER rotate the machine account password.
# The keytab in the vault is the single source of truth; rotation would
# desync the KVNO and break every subsequent boot. 0 = disabled.
ad_maximum_machine_account_password_age = 0

If it does desync — something ran realm join behind your back, or a well-meaning GPO rotated the password — the fix is the staging script itself: re-run it for that one host, and the adcli join resets the password while the upsert pushes the fresh keytab at the new KVNO. And if never rotating machine passwords makes you itch (it should, a little), the rotation job you'd want is just the staging script on a timer. I haven't needed it yet.

Running it

  • There is no unseal ceremony. restart: always plus the ENCRYPTION_KEY in .env means a power-cycled vault host comes back serving secrets with zero human intervention. Vault would sit sealed waiting for key shares — the right call for a bank, the wrong call for my garage.
  • Backup is one Postgres volume. pg_data is the entire vault state. Confession: the POC docs never actually wrote the backup step down. The honest runbook line is a pg_dump on a timer plus a copy of .env somewhere safe — the encryption key and the data are only useful together.
  • The vault only matters at boot. Running nodes hold their keytab in RAM and never re-fetch mid-uptime, so Infisical downtime affects exactly the nodes that reboot during it. The blast radius of "the keystore is down" is a queue of reboots, not an outage.
