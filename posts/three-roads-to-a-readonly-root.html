Three Roads to a Read-Only Root

July 2026 · POC writeup · full source in /poc

Five posts in this series have said "SUSE" like it was obvious. It wasn't. The repo still carries the fossils of the decision: two evaluation notes stamped LEGACY REFERENCE at the top, and a build script that refuses to run without zypper and admits, right in the error message, that "older Ubuntu/RHEL build logic was removed when the fleet standardized on SUSE." This is the story of that standardization — three distros, three house mechanisms for a root filesystem that forgets, and a winner chosen for reasons that mostly weren't the mechanisms.

What I was shopping for

The requirements were short. One golden image. A root that forgets writes — because even with RAM-boot as the primary path, the disk-deploy fallback for RAM-poor hardware needs statelessness too. And the smallest possible pile of distro-specific tooling to own, because every distro-flavored mechanism in the design is something I have to re-learn at every major version bump.

Ubuntu: overlayroot, the ergonomic winner

Ubuntu's answer is the overlayroot package, and credit where due — it has the best ergonomics of the three. One line: 

# /etc/overlayroot.conf
overlayroot="tmpfs:swap=1,recurse=0"

…or the same string as a kernel argument, and the initramfs mounts a tmpfs overlay on the read-only root before systemd ever runs. It even ships a maintenance escape hatch: overlayroot-chroot drops you into the real read-write disk underneath when you genuinely want a change to stick. Two wrinkles, though. The deployment lifecycle wants the first boot after imaging to be read-write so cloud-init can specialize the clone, with statelessness starting at boot two — solvable by moving specialization into the vault, which this design did anyway. And the wrinkle that actually mattered: Ubuntu's live and netboot world is casper and initramfs-tools, so the RAM-boot primary path means installing casper into the golden image and adopting its boot arguments. The best fallback mechanism of the three came attached to the most bespoke primary path.

RHEL: readonly-root, the enumerate-your-writes model

RHEL has no overlayroot package; its native pattern is philosophically different. An overlay says "write anywhere you like, it evaporates." RHEL's readonly-root says "writes fail unless the path is on the list": 

# /etc/sysconfig/readonly-root
READONLY=yes
TEMPORARY_STATE=yes

# /etc/rwtab.d/sssd-poc — the POC's whitelist
files  /etc/krb5.keytab
dirs   /var/lib/sss
dirs   /var/log/sssd

Everything in rwtab gets a tmpfs shadow; everything else is genuinely read-only. There's real rigor in that — the whitelist documents exactly what your system mutates, which the overlay model never forces you to learn. And to be fair on the primary path: dracut is a Red Hat project, livenet included, so RHEL RAM-boots the same way SUSE does, stock. What wore me down was SELinux. A keytab written at boot time by a script needs its file context handled, or SSSD gets denied reading it: 

semanage fcontext -a -t krb5_keytab_t /etc/krb5.keytab   # once, at image build
restorecon -v /etc/krb5.keytab                            # every boot, post-fetch

Not a blocker — two lines, written down in the notes. But it's a tax levied on every future change to the boot flow: each new file the fetch script touches is a fresh conversation with the policy. On a fleet whose entire identity model is "a script writes a credential into /etc at boot," that conversation never ends.

SUSE: the mechanism that isn't a distro feature

SUSE has no overlayroot either, and its answer barely counts as a SUSE answer: 

# one kernel argument — systemd does the rest
systemd.volatile=overlay

# wired into the disk-fallback image via grub:
GRUB_CMDLINE_LINUX_DEFAULT="systemd.volatile=overlay ..."
grub2-mkconfig -o /boot/grub2/grub.cfg

That's systemd's native volatile mode — a tmpfs OverlayFS upper layer assembled over the root in the initramfs, disk used read-only. No package, no distro patch, just upstream systemd behavior that SUSE doesn't get in the way of. The rest of the platform had the same texture: AppArmor's default SSSD profile already permits /etc/krb5.keytab, so the boot-time keytab write needs no relabeling ritual at all; pam-config --sss --mkhomedir wires PAM idempotently; and openSUSE Leap is binary-compatible with SLES 15, so the POC ran license-free on Leap with a straight face about flipping to registered SLES for production. SUSE charged its own tolls, to be clear: wicked — the default network stack, which wants a config file per interface — had to be disabled in favor of NetworkManager's match-all DHCP; SLES golden VMs need the SUSEConnect register-then-deregister ritual so the sealed image doesn't carry credentials; and SUSE names its initramfs /boot/initrd-<kver> where everyone else has an .img, a quirk the squashfs build script papers over with a three-way fallback.

The scorecard

Ubuntu RHEL SUSE volatile root overlayroot pkg readonly-root + systemd.volatile= (one conf line) rwtab whitelist overlay (kernel arg) live RAM-boot casper (add-on) dracut livenet dracut livenet (stock) (stock) keytab-at-boot AppArmor: SELinux: needs AppArmor: vs MAC security fine as shipped krb5_keytab_t fine as shipped PAM wiring pam-auth-update authselect pam-config

The real decider

Here's the thing the scorecard hides: all three fallback mechanisms were workable. If the volatile root had been the whole decision, I'd probably have shipped Ubuntu and its one-line overlayroot. The decision actually got made on the primary path, by a principle I only articulated afterwards: minimize the moving parts you own, and prefer the ones that are upstream plumbing rather than distro personality. Everything the RAM-boot design leans on — dracut livenet, systemd's volatile mode, NetworkManager, chrony, SSSD — is generic upstream infrastructure. SUSE was the distro that shipped all of it stock and then got out of the way: no casper bolted on, no SELinux policy conversation renewing itself every quarter. "Choosing SUSE" was really choosing to depend on plumbing that would look the same on any distro that carries it.

And the honest footnote: the Ubuntu and RHEL notes weren't deleted. They sit in the repo under their LEGACY banners, kept deliberately, because fleets are rarely pure — the day a mixed estate shows up, the rwtab whitelist and the overlayroot lifecycle notes are the head start. Standardizing on one road doesn't mean forgetting you mapped the other two.

All three evaluations live in 05-stateless-boot/ — start with suse-stateless-notes.md for the road taken, then ubuntu-overlayroot.md and rhel-readonly-root.md for the roads not. That wraps the roads-not-taken material from this POC; if your fleet walked one of the other two roads and it's working, I'd honestly love the counterexample.
