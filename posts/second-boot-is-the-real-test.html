The Second Boot Is the Real Test

July 2026 · POC writeup · full source in /poc

Three posts of architecture — the design, the vault, the boot pipeline — all making one claim: nothing persists. This last one is about how I decide whether that claim is true, and why the answer is never visible on the first boot. "It booted" turns out to be the least interesting fact about a boot.

The lies a first boot tells

Every failure mode I actually feared in this project is invisible on boot one. A box quietly leaking state onto a local disk looks identical to a stateless one, right up until the boot where the leak matters. A keytab that fetched fine today says nothing about whether the fetch path works every time or worked once. And the worst-case failure — something in the image running realm join behind my back and minting a duplicate AD computer object — literally cannot show up on the first boot, because the object count only moves on the boots after it.

The property I built is an invariant across reboots, and you can't verify an invariant with one sample. You need at least two, separated by the exact state transition that could break it. So the POC ships a verification script, and the script is designed around that reboot.

Eight checks, each hunting a specific lie

verify-poc.sh is about a hundred lines and runs as root on a deployed node. It isn't a smoke test — every check exists to catch a specific way the system could be lying to me. The overlay check catches a root filesystem that would silently persist writes. The time-sync check exists because Kerberos tolerates about five minutes of clock skew, and a failed kinit two checks later is unreadable without it. My favorite of the early ones is this pair: 

if grep -q 'root=live:' /proc/cmdline; then
  ok "booted via livenet: $(grep -o 'root=live:[^ ]*' /proc/cmdline)"
  grep -q 'rd.live.ram=1' /proc/cmdline \
    && ok "image fully copied to RAM (rd.live.ram=1)" \
    || bad "rd.live.ram=1 missing — node keeps a network dependency on the PXE server"
fi

That second line catches the subtlest failure in the whole design: a node that boots perfectly, runs perfectly, and has quietly kept a runtime dependency on the PXE server because the image never finished copying to RAM. Nothing about the running system looks wrong. The lie only surfaces the day the deploy server goes down and a "stateless" node hangs with it.

The identity checks climb a ladder of increasing proof. The keytab exists — fine. The keytab contains this host's principal — better, because "fetched a keytab" and "fetched the right keytab" are different claims when the vault is keyed by hostname and MAC. Then the real one: authenticate with it against the KDC. 

KRB5CCNAME=MEMORY: kinit -kt /etc/krb5.keytab \
  "$(echo "$HOSTN" | tr 'a-z' 'A-Z')\$@${REALM}"

A file on disk proves nothing; a ticket from the domain controller proves the machine account, the KVNO, and the clock all agree. Above that sit the user-facing layers — SSSD's domain reporting Online, an actual AD user resolving through id — and then the check that guards against my nightmare: adcli show-computer, run using the node's own keytab, confirming the computer object exists and there's exactly one of it. One design rule of the whole architecture is that only the staging script ever touches AD; this is the line that would catch a violation.

A small thing I'd defend in review: the script counts failures instead of dying on the first one — report first, gate at the end with [ "$FAIL" -eq 0 ]. On hardware, the second and third failures are usually the ones that tell you what the first one meant.

The canary

Check eight is the reason this post exists. Quoted nearly whole: 

echo "== 8. Statelessness canary =="
if [ -f /root/.poc-canary ]; then
  bad "canary from previous boot SURVIVED — root is NOT stateless"
  rm -f /root/.poc-canary
else
  touch /root/.poc-canary
  info "canary planted at /root/.poc-canary — reboot and re-run; step 8 must not FAIL"
fi

This check cannot pass on the run that plants it. It writes a file into the one place files aren't supposed to survive, and its only job is to be dead on arrival next boot. Which means the script's verdict is only meaningful the second time you run it — by design, the whole thing is a two-act play with a reboot for an intermission. And the canary is the check that keeps the other seven honest: if that file survives a reboot, then root isn't stateless, the keytab didn't come from the vault this boot, and every green PASS above it is describing a machine I only think I have.

boot 1 stream image → RAM → fetch keytab → verify-poc.sh: all PASS plant canary: /root/.poc-canary proves: the happy path works — once ↓ reboot (entire OS discarded from RAM) boot 2 stream image → RAM → fetch keytab → verify-poc.sh again canary gone? → root really is stateless keytab re-fetched? → the vault path works every boot, not one boot AD objects still 1? → nothing joined behind my back proves: the invariant

Acceptance is a sentence, not a dashboard

The script ends by printing its own pass criterion, and I've grown attached to having it there, where the person staring at a serial console at 2am will actually read it: 

RESULT: 11 passed, 0 failed
POC acceptance = two consecutive clean runs across a reboot with FAIL=0.

The implementation guide tightens it further: the AD object count for a host stays at exactly one across at least three reboots, and before any fleet rollout the script runs on one unit of each hardware model (plus a dmesg -l err sweep), because a fleet of mismatched hardware fails per-model, not per-fleet. Gaps get fixed by adding a driver to the single image — never by forking a per-model one.

The general rule

None of this is specific to PXE or Kerberos. The pattern is: when a system claims a property, don't test the demo — write the check that would catch the system lying about the property, and put the state transition that could break it inside the acceptance criterion. "Stateless" is a claim about reboots, so the reboot goes in the test. "Highly available" is a claim about failures, so the failure goes in the test. A first boot, a first request, a first failover — the first anything mostly measures whether you wrote a demo. The second one measures whether you built what you said you built.

The script is 07-verification/verify-poc.sh in the POC repo — a hundred lines, no dependencies beyond what the image already carries. Steal the canary trick; it's the cheapest honest test I've ever written. And that wraps this series — if you've read all four and still think the design holds, go run it twice and prove me wrong on the second one.

Thanks for reading. This came out of a working proof-of-concept — go break it and tell me what I missed. Source
